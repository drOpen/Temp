This document describes user stories for DrTest
Story Description Status
1. Каждый тест самодостаточен, содержит в себе всю информацию по запуску, выполненным шагам и их статусу Proposed
2. Тест может содержать неограниченное количество маркеров 'Inclusions', которые формируются на основании маркеров его дитей: как действий самого теста, так и сабтестов, если это тест - контейнер Proposed
3. Может ли тест содержать в себе сабтесты или вводится отдельная сущность тестовый контейнер. Например, ввести понятие Tests Proposed
4. Тест должен содержать уникальный идентификатор запуска. Этот идентификатор присваивается всем тестам, контейнерам тестов, запускаемым совместно, не зависимо от того, где и когда происходит сам запуск. В дальнейшем этот ID позволит найти все тесты из этого запуска. Proposed
5. Необходимо обеспечить поддержку, идентификации теста, который клонировался - копировался и запускался на одной/разной машине. Например, есть описанные тесты, которые в процессе могут копироваться и выполняться на разных машинах, это один запуск id, это один тест, у обоих клонов парент id одинаков, но это разные результаты одного и того же теста! Нужен идентификатор! Который позволит посмотреть результат выполнения одного и того же теста в рамках одного запуска на разных компьютерах, лабораториях, повторный запуск на одной и той же машине! Proposed
5. Необходимо обеспечить поддержку, с возможностью восстановления - построения в отчете исходного дерева тестов. Например, есть исходное описанные тестов. Должна быть возможность восстановить эту иерархию только на основании результатов присланных тестов. Например, через parent id Proposed
6. Обеспечить однозначную идентификацию исходного теста и всю его историю. Для чего, вводится понятие TestID - TID. TestID это уникальный идентификатор каждого теста в виде GUID, группы тестов, который присвается ему при создании автором теста и никогда не меняется, включая клонирование теста. Таким образом всегда можно посмотреть историю выполнения данного теста на различных билдах, на различных системах и прочее. Необходимо понимать, что TestID не уникален в рамках SessionID и может встречать много раз за один запуск группы тестов. Proposed
7. Требуется обеспечить связь всех тестов в рамках запуска всего комплекса тестов, - сессии. Для чего, вводится понятие SessionID - SID. Все тесты в рамках одного старта, сессии, имеют одинаковый SessionID который выставляется test launcher'ом, который отвечает за начальную компиляцию, агрегацию и запуск тестов. Proposed
8. Необходимо предусмотреть формат данных, передающий информацию о сессии, например, номер билда, комментарий к запуску, и прочее. Для обеспечения связи между тестами, набором тестов, и свойст сессии требуется использовать SessionID. Proposed
9. Требуется различать один и тот же тест запущенный в одной сессии дважды, например, на разных тестовых рабочих станциях, для чего вводится понятие ExecutionID - EIT. ExecutionID выставляет непосредственно в момент запуска скомпилированного теста. Делает это конечный launcher. Пример взаимосвязи описанных выше идентификаторов: один запуск одного теста на разных рабочих системах, приведет к одинаковым SeesionID и TestID, но разным ExecutionID. Proposed
10. Необходимо отличать во времени старые и новые данные одного и того же теста, для чего вводится ChangeID - CID Каждый тест имеет ChangeID изначально равный 0 и инкрементирующийся на единицу при любых изменениях теста при его запуске. Увеличение ChangeID вносится при сохранении любых изменений в тесте или группы тестов непосредственно тем компонентом который производит сохранение состояния теста. Proposed
11. Система хранения и отображения статситики запуска тестов должна предуматривать асинхронную работу с потоком тестов:
11a. Тесты могут приходить от разных сессий. Каждый тест имеет отличительный SessionID, на основании которого можно отличить один запуск группы тестов от другого. Proposed
11b. Статус одного теста может приходить не в хронологической последовательности. Каждый запуск теста можно четко определить на основании уникальности ExecutionID, поэтому тест находится по этому идентификатору и обновляется только с меньшего на больший идентификатор изменения - ChangeID. Proposed
